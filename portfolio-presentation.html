Un experimento de optimización de flujos de desarrollo donde modelos fundacionales avanzados generan código bajo dirección humana estricta, y donde yo asumo el rol de Product Owner: arquitectura, métricas de calidad, decisiones de UX y validación contra datos reales. QA · Dirección de producto

Logs de dirección · Human-in-the-loop

El modelo propone, yo decido. Casos reales donde corregí al asistente porque la lógica generada era metodológicamente incorrecta o visualmente tosca. Algoritmo de estancamiento dinámico y rediseño de la matriz de frecuencia de 12 semanas (Apple-style heatmap) incluidos.

Estancamiento dinámico bug lógico

El asistente proponía subir el peso por defecto en la próxima sesión aunque el usuario no hubiera cumplido el objetivo. Resultado: sugerencias inseguras tras una sesión floja, riesgo de lesión y de falsos PRs.

Reescribí la regla a "double progression estricta" con detección de estancamiento sobre 1RM estimado en ventana de 4 semanas: solo sube si TODAS las series del top del rango se cumplen, y nunca baja por inercia.

// stagnation.js · ventana de 4 semanas
const recent = sessions.filter(s => s.date >= cutISO);
return bestEstimated1RM(recent) <= bestEstimated1RM(before);

Matriz de frecuencia 12 sem. rediseño UI

La matriz original se veía tosca y desalineada: niveles por volumen mezclado, sin etiquetas de día ni mes, semanas no alineadas a lunes. No comunicaba.

Apliqué el patrón de los widgets de Apple: 12 semanas L→D alineadas a lunes, niveles por nº de ejercicios distintos, opacidades 22 / 45 / 70 / 95 %, etiquetas "L M X J V S D" a la izquierda, meses en el eje superior, leyenda discreta a la derecha.

// volume.js — adherenceMatrix()
const lvl = count <= 0 ? 0 : Math.min(4, count);
// future cells: opacity .4 · pointer-events: none

Timer real-clock PWA hardening

El timer de descanso usaba setInterval-as-counter. iOS lo throttlea al bloquear pantalla → al volver, marcaba 2:00 cuando realmente habían pasado 4:00.

Reescritura con timestamp objetivo (endAt) y recálculo desde Date.now() en cada tick + visibilitychange. Sobrevive al bloqueo de pantalla entre series.

this.endAt = Date.now() + seconds * 1000;
this.remaining = Math.round((this.endAt - Date.now()) / 1000);

Service Worker deploy trap

Cache-first sobre el HTML dejaba a los usuarios congelados en una versión instalada en pantalla de inicio: un deploy nuevo NO llegaba nunca.

Reescribí la estrategia: network-first para HTML, stale-while-revalidate para JS/CSS, y el propio sw.js NUNCA se intercepta. Toast de "nueva versión — recarga" sin auto-reload (no interrumpir una serie).

if (url.pathname.endsWith('/sw.js')) return; // nunca cachear
if (isNav) return networkFirst(req);
